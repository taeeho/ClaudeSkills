# wiki-add

theowiki(`~/Desktop/theowiki`)에 시행착오·노트를 정해진 형식으로 저장한다.

> 경로 안의 `~` 은 OS에 맞춰 expand:
> - PowerShell: `$env:USERPROFILE` (예: `C:\Users\MKM10011`)
> - Bash/zsh: `$HOME` (예: `/Users/yourname`)
> 도구 호출 시점에 절대경로로 변환할 것.

## 트리거
- "위키에 넣어줘"
- "wiki에 저장해줘"
- "이거 기록해둬"
- "/wiki-add"

## 입력 (이 중 하나)
1. 직전 대화 맥락 (현재 세션에서 다뤘던 문제·해결)
2. 사용자가 별도로 붙여넣은 텍스트·로그·스크린샷
3. 사용자가 args로 짧게 요약 ("아까 N8N webhook 401 건 정리해줘")

## 작업 절차

### Step 0: 가치 판정 (Judge) — 저장 전 필수
raw에 무조건 던지지 않는다. 다음 3기준으로 먼저 자체 평가:

1. **재발 가능성**: 6개월 안에 또 마주칠 일인가?
   (1회성 환경 이슈·우연한 typo는 ❌)
2. **비자명성**: 공식 문서/구글링 1분이면 나오는가?
   (그러면 검색하면 됨, 위키 노이즈만 늘림)
3. **맥락 의존성**: AX기획팀 / 이 회사 스택 / 이 사용자만의 노하우인가?
   (회사·스택 특수 사항이면 무조건 GO)

판정 매트릭스:
| 조건 | 판정 |
|------|------|
| 재발 + 비자명 + 맥락의존 (3개 만족) | **반드시 ingest** |
| 맥락의존만 만족 | **ingest 권장** (회사 노하우는 검색 안 됨) |
| 재발만 만족 | **고민 후 결정** — 사용자에게 짧게 묻기 |
| 비자명만 만족 / 0개 만족 | **스킵 권장** |

❌ 스킵 권장이면 **저장하지 말고** 사용자에게 보고:
```
⏭️  이 내용은 wiki에 안 넣는 걸 추천:
  - 재발 가능성: 낮음 (1회성 환경 이슈)
  - 비자명성: 낮음 (공식 문서 첫 페이지에 있음)
  - 맥락 의존성: 낮음 (이 회사·스택 특수 사항 아님)
  그래도 저장할까? (y/n)
```

y면 강제 진행, n/무응답이면 종료.
✅ 통과면 Step 1로.

**원칙**: "raw는 적게, wiki는 깊게."
양으로 승부하지 않는다. 한 주제에 깊이 쌓이는 페이지가 좋은 페이지.

### Step 1: 내용 파악 + 부족한 정보 질문
대화·입력에서 다음을 추출:
- **증상** (무슨 일이 있었나, 에러 메시지)
- **맥락** (어떤 워크플로우/코드/상황)
- **시도한 것들** (시행착오 단계)
- **원인·해결** (resolved인 경우)
- **관련 도구/주제** (tag 추출용)

부족한 핵심 정보가 있으면 **저장 전에** 사용자에게 1~2개만 짧게 질문.
- 예: "이거 resolved야 아니면 workaround야?", "tag는 [n8n, webhook] 맞아?"
- 절대 추측으로 채우지 말 것 — TBD 또는 질문.

### Step 2: 형식 결정
입력 성격에 따라 4가지 중 선택:

**A. 시행착오 (기본 — 대부분 이 형식)**
```markdown
---
date: YYYY-MM-DD
tags: [도구, 주제]
status: resolved | open | workaround
severity: low | medium | high
related: []
---

# 증상
(에러·재현 단계)

# 맥락
(어디서, 뭘 하다, 언제)

# 시도한 것들
1. ❌ A → 결과
2. ✅ B → 됨

# 원인
# 해결책
# 교훈 (한 줄)
```

**B. 짧은 발견·노트** (시행착오까진 아닌 작은 깨달음)
```markdown
---
date: YYYY-MM-DD
tags: [도구, 주제]
---

(2~5줄짜리 발견·팁)
```

**C. 코드 스니펫**
```markdown
---
date: YYYY-MM-DD
tags: [도구, snippet]
type: snippet
---

# (스니펫 제목)

```언어
(코드)
```

# 언제 씀
- ...
```

**D. 클리핑 아티클** (사용자가 URL·아티클 본문 제공)
```markdown
---
source: URL
date_clipped: YYYY-MM-DD
tags: [주제]
---

(본문)
```

분기 기준:
- 에러·해결 시도가 있음 → A
- 코드 블록이 핵심 → C
- URL·외부 글 → D
- 그 외 짧은 메모 → B

### Step 3: 파일명 생성
**규칙**: `YYYY-MM-DD-짧은-keyword.md`
- 소문자, 단어는 하이픈 연결, 한글 가능하나 공백 대신 하이픈
- 키워드는 검색 가능한 명사 위주 (`bug.md` 같은 모호한 이름 금지)
- 예: `2026-05-04-n8n-webhook-401.md`, `2026-05-04-supabase-rls-anon-null.md`

날짜는 오늘 날짜 (글로벌 CLAUDE.md의 `currentDate` 또는 시스템 날짜).

### Step 4: 저장 위치 결정
- A, B → `raw/trial-and-error/`
- C → `raw/trial-and-error/` (snippet도 여기, type 필드로 구분)
- D → `raw/articles/`

### Step 5: 파일 생성
Write 도구로 `~/Desktop/theowiki/raw/<폴더>/<파일명>` 생성 (OS에 맞춰 절대경로로 expand).

### Step 6: 사용자에게 확인 + ingest 제안
저장 후 사용자에게 보고:
```
✅ 저장: raw/trial-and-error/2026-05-04-n8n-webhook-401.md

지금 ingest해서 wiki/ 페이지까지 정리할까? (y/n)
```

사용자가 yes면 → theowiki의 CLAUDE.md에 정의된 ingest 워크플로우 실행:
1. wiki/sources/ 에 요약 페이지 생성
2. 관련 entities/topics 페이지 업데이트 (없으면 생성)
3. wiki/index.md 업데이트
4. wiki/log.md append: `## [YYYY-MM-DD] ingest | 제목`

## 추가 규칙

### 보안 체크
저장 전 다음이 포함됐으면 **사용자에게 경고**하고 마스킹 제안:
- API 키, 토큰, 비밀번호
- 실제 고객 이메일·전화번호·식별자
- 회사 내부 URL·서버 IP

### 추측 금지
- 대화에 없던 원인·해결책을 지어내서 채우지 말 것
- 모르면 `TBD` 또는 빈 섹션으로 두고 사용자에게 질문
- "교훈 (한 줄)"도 명확하지 않으면 사용자에게 물어보기

### 1파일 = 1사건
대화에서 여러 별개 이슈가 나왔으면 **여러 파일로 분리** 저장.
한 파일에 섞어 넣지 말 것.

### 중복 체크
저장 전 `raw/trial-and-error/` 에 비슷한 파일명·주제가 있는지 Glob으로 확인.
있으면 사용자에게: "기존 파일 X에 추가할까, 새 파일 만들까?" 질문.

## 예시 흐름

**사용자**: "아까 그 N8N OAuth2 401 건 위키에 넣어줘"

**Claude 작업:**
1. 대화 스크롤백에서 N8N OAuth2 401 관련 내용 추출
2. 부족 정보 확인 → tag 확인 질문 ("tags: [n8n, webhook, oauth] 맞아?")
3. 형식 A로 마크다운 작성
4. 파일명: `2026-05-04-n8n-oauth2-401.md`
5. 중복 Glob 체크 → 없음
6. Write 실행
7. "✅ 저장 완료. ingest할까?" 확인
8. y → ingest 워크플로우 실행
